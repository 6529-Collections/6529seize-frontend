# PR Report: Active Wave Reaction Read

PR: https://github.com/6529-Collections/6529seize-frontend/pull/2787
Branch: codex/wave-reaction-active-read -> main
Related PRs: None
Mode: staging-deploy

## Summary
Realtime reaction updates in an open wave now reuse the same active-wave read path as new drop updates only when the reacted drop is already near the wave viewport, so reactions the user is likely seeing are treated as read without suppressing notifications for loaded but offscreen drops.

## What Changed
- Marked the active visible wave as read after a `DROP_REACTION_UPDATE` websocket event is reconciled for a loaded drop.
- Gated reaction auto-read behind the existing virtual-scroll `isInView` signal for the reacted drop.
- Reused the existing delivered-notification cleanup and wave notification read API call.
- Added focused regression tests for near-viewport and offscreen reaction read behavior.

## Backend Impact
Not affected.

## Frontend Impact
- Routes/views: Wave chat and wave drop realtime views.
- Components: `useWaveRealtimeUpdater` active-wave read behavior.
- Generated API/client: Not affected.
- User-visible behavior: Users actively viewing a wave should no longer receive push notifications for reaction updates on drops already near their viewport, while reactions on offscreen loaded drops remain eligible for notification.
