# PR Report: Restore Wallet Auth Session V2 Client Hardening

PR: https://github.com/6529-Collections/6529seize-frontend/pull/2843
Branch: `codex/restore-wallet-security-hardening` -> `main`
Related PRs: https://github.com/6529-Collections/6529seize-backend/pull/1616
Mode: `pr-ready`

## Summary

Frontend auth v2 support for structured wallet signatures, web/native session-v2
flows, connection sharing, websocket/session compatibility, and v1-to-v2
migration prompts driven by backend settings.

## What Changed

- Added frontend session-v2 login/refresh/logout utilities and v2 session
  marker persistence.
- Added structured wallet-signature client paths and generated API model updates
  from the backend auth contract.
- Added connection sharing acceptance flow and web-to-native handoff support.
- Mobile/native connection-share URLs omit role metadata. The client accepts
  any old mobile/native `role` URL parameter harmlessly, but redeem persists
  the server-returned role as authoritative.
- Desktop connection sharing remains the temporary legacy exception: the
  frontend either reuses the active account's legacy refresh token or calls
  `/auth/connection-share/legacy-desktop` to build a Desktop deep link with
  `token`, `address`, and optional `role`.
- The share modal always renders the Connection tab. Legacy-authenticated users
  see an Update Authentication action instead of a QR/link, and that action
  starts the v2 auth upgrade flow.
- Replayed the local v1-auth migration prompt behavior: 2-hour reminders,
  72-hour local grace capped by backend deadline, shared/non-signable auth
  reshare guidance, and cleanup after expiration.
- Simplified cookie-backed web auth config: the FE now sends session-v2
  credentials to the configured `API_ENDPOINT`; the old
  `WEB_SESSION_CREDENTIAL_API_ORIGINS` env is no longer read.
- Refreshed OpenAPI/generated models from the updated backend branch after
  merging latest `main`.
- Restored the existing web connected-profile workflow: session-v2 web users
  can still add connected profiles up to `MAX_CONNECTED_PROFILES` instead of
  being treated as a single-profile cookie session.
- Preserved active stored session-v2 JWTs when the web refresh cookie belongs
  to a different connected profile, so switching profiles does not immediately
  force another signature while the active profile JWT is still valid.
- Short-circuited validation for already-valid active session-v2 JWTs before
  refreshing the web cookie. Current web refresh/logout calls also send
  `client_address`, so expired-token refreshes and logout target the matching
  address-scoped backend cookie instead of whichever account last wrote the
  compatibility cookie.
- Connection share creation now relies on the backend session-v2 check instead
  of a local v2 marker, so a successfully upgraded web session is not blocked by
  stale browser-side auth state.
- The share modal now attempts backend share creation whenever there is an
  active wallet address and regenerates the one-time share code when the active
  wallet changes while the modal is open.
- Aligned wallet-auth contract docs, API-tool copy, share-modal docs, and public
  tech-report copy with the final session-v2 connection-sharing terminology and
  deep-link shape.
- Session upgrade prompts stay mounted while auth validation reruns, preventing
  modal flicker during refresh/validation loops.
- For disconnected active-profile upgrade prompts, the client disconnects the
  currently live wallet before opening the connect flow. This lets users switch
  from a still-connected wallet B back to authenticated-only wallet A without
  hitting the AppKit "already linked" state.
- Added explicit coverage that session-v2 migration reminder dismissals are
  scoped per connected account.

## Frontend Impact

- Routes/views: connection sharing acceptance route and auth migration prompts
- Components: auth providers/context, wallet setup/adapter behavior,
  header/sidebar auth display paths, connected-profile add/switch behavior,
  migration modals
- Generated API/client: auth settings, session-v2, native refresh, connection
  sharing, and latest main OpenAPI additions
- User-visible behavior: users can migrate v1 auth to v2 with a controlled grace
  period once backend settings enable it; new session-v2 flows use cookie-backed
  web auth through `API_ENDPOINT`; connection sharing clearly prompts legacy
  users to update auth first

## Config / Env

- `API_ENDPOINT`: frontend, required. Production:
  `https://api.6529.io`; staging: `https://api.staging.6529.io`.
- `WEB_SESSION_CREDENTIAL_API_ORIGINS`: frontend, deprecated and removed from
  the public env schema/sample. It is no longer read.
- `SESSION_V2_MIGRATION_DEADLINE`: backend-owned only; frontend reads it from
  `/api/settings.auth`.
- `AUTH_STRUCTURED_SIGNATURES_REQUIRED`: backend-owned only; frontend reads it
  from `/api/settings.auth`.
- Backend web-origin config is handled by backend `WEB_APP_ORIGIN` /
  `WEB_APP_ADDITIONAL_ORIGINS`; no matching FE env is required.
- Native client rollout is a release gate for cutoff phases. The web deploy can
  be silent, but strict signatures and legacy-refresh shutdown should wait until
  iOS/Android builds with session-v2 native refresh and connection-share redeem
  behavior are available and verified.

## Release Notes

- Deploy backend first with strict flags off and
  `SESSION_V2_MIGRATION_DEADLINE` unset, then deploy this frontend. Set the
  backend deadline only when migration prompts should begin.
- If native apps are released separately from the web frontend, verify the
  native release candidates before enabling strict signatures or disabling
  legacy refresh.
- No separate FE credential-origin env is needed for cross-origin web auth; the
  configured `API_ENDPOINT` is the frontend source of truth.
- Existing v1 users are not forced out by frontend deploy alone. Backend
  settings control when migration prompts and cutoffs start.
- Existing connected-profile UX is preserved for web session-v2. `Add` remains
  available until the normal connected profile limit is reached.
- Switching between stored connected profiles does not depend on the currently
  connected browser wallet when the target profile still has a valid v2 JWT.
- Connection sharing remains available unless disabled on the backend, but share
  creation requires v2 web auth. Legacy-authenticated users can update auth
  directly from the Connection tab.
- Desktop connection sharing remains available through the legacy Desktop
  handoff while backend legacy refresh redemption remains enabled.
- The session-upgrade modal remains stable during validation reruns, and its
  connect action now clears the currently live wallet connection before asking
  the user to reconnect the active profile wallet.

## Risks / Follow-Ups

- Do not enable backend strict signature mode or legacy-refresh shutdown until
  this frontend is deployed and migration readiness checks for known web,
  native, and external clients are complete.
