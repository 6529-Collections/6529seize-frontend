# PR Report: Drop Forge claim path split

PR: https://github.com/6529-Collections/6529seize-frontend/pull/3226
Branch: codex/tier-f-drop-forge-split -> main
Related PRs: Parent tracker https://github.com/6529-Collections/6529seize-frontend/pull/3111; prior Tier D/E integration https://github.com/6529-Collections/6529seize-frontend/pull/3206

## Summary
Splits the Drop Forge craft and launch claim clients into local sections, view modules, state/effect hooks, and write helpers while preserving the existing public entry files, route behavior, hook order, wallet/auth/session handling, transaction flows, copy, and UI states.

## What Changed
- Kept the craft and launch claim page clients as stable facades and moved their existing implementation blocks into feature-local modules.
- Kept launch presentation modules hook-free and preserved the original state, effect, view-model, modal-action, and write ordering.
- Preserved the initialize/update claim, metadata-only update, airdrop, research airdrop, pay-artist, media editing, Arweave, and distribution flows.
- Lowered the committed oversized-file baseline from 42 on main to 38 on the combined branch. Three removals are directly attributable to this PR; the fourth locks in an already-present main-branch reduction for `helpers/waves/wave-rules.helpers.ts`.

## Frontend Impact
- Routes/views: `/drop-forge/craft`, `/drop-forge/craft/{id}`, `/drop-forge/launch`, and `/drop-forge/launch/{id}` retain their existing contracts.
- Components: Drop Forge craft claim editor and launch claim operations internals.
- Generated API/client: not affected.
- User-visible behavior: intended to be unchanged.

## Config / Env
No config or environment changes.

## Release Notes
Under-the-hood maintainability work for Drop Forge claim preparation and launch operations. Operators should see the same claim editing, publishing, phase configuration, airdrop, payment, wallet, session, and transaction behavior.

## Risks / Follow-Ups
The residual risk is extraction wiring around a critical operational path. Staging validation should cover:
- `/drop-forge/craft` list loading and navigation into a real `/drop-forge/craft/{id}` claim.
- Craft image and animation upload/link/remove/revert/save flows.
- Craft season, edition size, traits, description, external URL, Arweave publishing, distribution summaries, and photo lightbox behavior.
- `/drop-forge/launch` list loading and navigation into the matching `/drop-forge/launch/{id}` claim.
- Launch phase tab selection, initialize/update validation, metadata-only updates, airdrop and research-airdrop preparation, pay-artist preparation, and transaction modal states.
- Permission loading/denial, wallet disconnect/reconnect, session refresh, chain selection, and the Sepolia testnet indicator/contract path.
