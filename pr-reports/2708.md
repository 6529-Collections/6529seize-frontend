## Summary

Adds REP category discovery and detail pages for Profile REP and Wave REP analytics, then hardens client caching so viewer-specific Wave REP data is not reused across auth or proxy context changes.

## What Changed

- Added `/rep/categories` and `/rep/categories/[category]` with search, suggested categories, deep links, Profile REP tables, and Wave REP tables.
- Added Network/header discovery for REP Categories.
- Shows category-wide Wave REP waves and contributors without requiring a separate wave search.
- Displays wave/profile avatars and existing profile hover-card links in analytics rows.
- Scopes the three Wave REP React Query keys by `viewerProfileId`, `proxyId`, and whether the active proxy has `READ_WAVE`, preventing private wave analytics cached for one viewer from being shown during another viewer/proxy context.
- Added query-key unit coverage for the viewer visibility scope.

## Frontend Impact

- Routes/views: `/rep/categories`, `/rep/categories/[category]`.
- Components: REP category explorer/detail, Wave REP analytics scope, navigation/search entry points.
- Generated API/client: frontend OpenAPI and generated models already refreshed from backend PR #1640 earlier in this PR.
- User-visible behavior: users can browse category-level Profile REP and Wave REP analytics; Wave REP rows are partitioned by the current viewer/proxy visibility context.
