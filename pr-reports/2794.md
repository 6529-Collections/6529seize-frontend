# PR Report: Native Profile Overlay Navigation

PR: https://github.com/6529-Collections/6529seize-frontend/pull/2794
Branch: codex/native-screen-stack-swipe -> main
Related PRs: None
Mode: pr-ready

## Summary
Adds a Capacitor-only native profile navigation stack for profile links opened from waves or messages. The replacement avoids App Router route interception entirely, so web and mobile-web navigation keep their normal route behavior.

## What Changed
- Removed the App Router `@nativeOverlay` implementation from `/waves` and `/messages`.
- Added a client-side `NativeProfileNavigationProvider` that runs only when `useDeviceInfo().isApp` is true.
- Added a document-level profile-link interceptor scoped to waves/messages source routes and same-origin profile destinations.
- Added a bounded native profile stack with a maximum of 3 entries, native history state, browser Back/Forward support, and inward edge-swipe close.
- Preserved the original waves/messages location in the native app shell while the overlay URL is visible, so the background stream stays mounted.
- Added pure helper tests for source-route detection, profile target parsing, and history stack bounds.

## Frontend Impact
- Routes/views: no App Router route files are added; native app profile-link taps from waves/messages update history and show an overlay without route navigation.
- Components: `NativeProfileNavigationProvider`, native profile navigation helpers, and `AppLayout` background-location handling.
- Generated API/client: Not affected.
- User-visible behavior: in the Capacitor app only, profile links opened from waves/messages render in a native overlay on top of the preserved stream; Back/Forward and inward edge swipe move through or close the overlay stack.

## Config / Env
None.

## Release Notes
Native app users can inspect profiles from waves or messages without losing their place, then close the profile with browser Back or an inward edge swipe.

## Risks / Follow-Ups
- The previous App Router intercepted-route approach was rolled back from staging and replaced because it could not be truly Capacitor-only.
- This implementation intentionally renders profile identity/header content in the overlay instead of embedding full server-rendered profile subpages. Full subpage parity would require a larger client-owned profile surface.
