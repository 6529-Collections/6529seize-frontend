# PR Report: Wallet Auth Session V2 Client Hardening

PR: https://github.com/6529-Collections/6529seize-frontend/pull/2556
Branch: `codex/wallet-security-hardening` -> `main`
Related PRs: https://github.com/6529-Collections/6529seize-backend/pull/1616
Mode: `pr-ready`

## Summary

Frontend auth v2 support for structured wallet signatures, web/native session-v2
flows, connection sharing, websocket/session compatibility, and v1-to-v2
migration prompts driven by backend settings.

## What Changed

- Added frontend session-v2 login/refresh/logout utilities and v2 session
  marker persistence.
- Added structured wallet-signature client paths and generated API model updates
  from the backend auth contract.
- Added connection sharing acceptance flow and web-to-native handoff support.
- Replayed the local v1-auth migration prompt behavior: 2-hour reminders,
  72-hour local grace capped by backend deadline, shared/non-signable auth
  reshare guidance, and cleanup after expiration.
- Simplified cookie-backed web auth config: the FE now sends session-v2
  credentials to the configured `API_ENDPOINT`; the old
  `WEB_SESSION_CREDENTIAL_API_ORIGINS` env is no longer read.
- Refreshed OpenAPI/generated models from the updated backend branch after
  merging latest `main`.
- Restored the existing web connected-profile workflow: session-v2 web users
  can still add connected profiles up to `MAX_CONNECTED_PROFILES` instead of
  being treated as a single-profile cookie session.
- Preserved active stored session-v2 JWTs when the web refresh cookie belongs
  to a different connected profile, so switching profiles does not immediately
  force another signature while the active profile JWT is still valid.
- Short-circuited validation for already-valid active session-v2 JWTs before
  refreshing the single web cookie. This keeps switching back to Account A
  working while the browser wallet/web cookie is still on Account B.
- Added explicit coverage that session-v2 migration reminder dismissals are
  scoped per connected account.

## Frontend Impact

- Routes/views: connection sharing acceptance route and auth migration prompts
- Components: auth providers/context, wallet setup/adapter behavior,
  header/sidebar auth display paths, connected-profile add/switch behavior,
  migration modals
- Generated API/client: auth settings, session-v2, native refresh, connection
  sharing, and latest main OpenAPI additions
- User-visible behavior: users can migrate v1 auth to v2 with a controlled grace
  period once backend settings enable it; new session-v2 flows use cookie-backed
  web auth through `API_ENDPOINT`

## Config / Env

- `API_ENDPOINT`: frontend, required. Production:
  `https://api.6529.io`; staging: `https://api.staging.6529.io`.
- `WEB_SESSION_CREDENTIAL_API_ORIGINS`: frontend, deprecated and removed from
  the public env schema/sample. It is no longer read.
- `SESSION_V2_MIGRATION_DEADLINE`: backend-owned only; frontend reads it from
  `/api/settings.auth`.
- `AUTH_STRUCTURED_SIGNATURES_REQUIRED`: backend-owned only; frontend reads it
  from `/api/settings.auth`.
- Backend web-origin config is handled by backend `WEB_APP_ORIGIN` /
  `WEB_APP_ADDITIONAL_ORIGINS`; no matching FE env is required.

## Deployment

No staging or production deployment was executed by request.

Planned frontend order when approved:

1. Deploy backend PR first with strict flags off and
   `SESSION_V2_MIGRATION_DEADLINE` unset.
2. Deploy frontend with only the normal `API_ENDPOINT` value for the target
   environment.
3. Smoke test session-v2 login, refresh/logout, connection sharing, websocket
   behavior, and v1 migration prompts.
4. Set backend `SESSION_V2_MIGRATION_DEADLINE` when prompts should begin.
5. Later enable backend strict signature mode and legacy-refresh shutdown after
   migration readiness checks.

## Release Notes

- No separate FE credential-origin env is needed for cross-origin web auth; the
  configured `API_ENDPOINT` is the frontend source of truth.
- Existing v1 users are not forced out by frontend deploy alone. Backend
  settings control when migration prompts and cutoffs start.
- Existing connected-profile UX is preserved for web session-v2. `Add` remains
  available until the normal connected profile limit is reached.
- Switching between stored connected profiles does not depend on the currently
  connected browser wallet when the target profile still has a valid v2 JWT.
- Connection sharing remains available unless disabled on the backend.
