# Museum Static-Page Migration Feasibility Report

Generated by `scripts/profile-cms/migrate-static-pages.ts` against the
`scripts/profile-cms/manifests/museum.manifest.json` manifest (all 143
`app/museum/**/page.tsx` files). This report is the deliverable for the
museum pilot; the generated 143-page package is **not** committed to the
repo (see "Chunking recommendation" below for why one giant package is not
the right shape anyway) and does not need to validate clean.

Raw per-page/warning/asset-host data backing this report is in
`scripts/profile-cms/reports/museum-migration-raw.json` (git-ignored-scale
working artifact regenerated by rerunning the converter; not committed).

converter could not map to a typed block was either preserved in a
`rich_text` fallback block or explicitly recorded as a warning naming the
untouched source content (video/iframe URL).

## Validation result

Running `validateCmsPackageV1(pkg, { allowFixtureSignatures: true,
allowFixtureStorage: true, enforceHashes: true })` against the generated
143-page museum package returns **`valid: true`, 0 issues**.

This is expected, not a sign the migration is complete: the converter is
deliberately conservative and only emits blocks/asset references it can
build safely (https-only image sources, safe button hrefs, valid
width/height on every image asset), so nothing it emits fails schema or
semantic validation. The **gap is coverage, not correctness** — see
"Unconverted patterns" for what a validator will never catch because the
converter chose not to emit it in the first place (video/iframe embeds).

**Every single image asset referenced across all 143 museum pages resolves
to `dnclu2fna0b2b.cloudfront.net`** — the legacy WordPress/Avada CloudFront
distribution. This is a **different host** from
`PROFILE_CMS_ASSET_PROXY_ALLOWED_HOSTS` in
`lib/profile-cms/runtime/mediaProxy.ts`, which only allows
`d3lqz0a4bldqgf.cloudfront.net` with path prefixes `/6529-emoji/` and
`/images/`. Consequences:

- `dnclu2fna0b2b.cloudfront.net` asset URIs pass the protocol-level safety
  check (`CMS_SAFE_URI_PROTOCOLS` only checks the scheme is `https:`/
  `ipfs:`/`ar:`/`arweave:`, which schema/semantic validation enforces), so
  `validateCmsPackageV1` does not flag them.
- They will **not** be proxied by `isProfileCmsAssetProxyAllowedUrl` /
  `getProfileCmsAssetProxyUrl` — that function only rewrites URLs on the
  allowlisted host+path-prefix combination and passes everything else
  through unchanged (`getProfileCmsAssetProxyUrl` returns the input
  verbatim when not allowed, it does not error). Whether that matters
  depends on whether `dnclu2fna0b2b.cloudfront.net` needs proxying at all
  (CORS/CSP) for the CMS renderer to load these images directly.
- `videos.files.wordpress.com` (self-hosted video sources, 312 references
  across 34 pages) and `generator.artblocks.io` (Art Blocks generator
  iframes, 11 references across 3 `6529-fund-szn1` pages) are two more
  hosts entirely outside any current CMS allowlist, and unlike the images
  they are not converted to typed blocks at all yet (see below) — so they
  are not just unproxied but currently un-migrated.

**Recommendation:** before a real (non-pilot) museum migration, either (a)
add `dnclu2fna0b2b.cloudfront.net` (and, if self-hosted video/iframe
support is in scope, `videos.files.wordpress.com` and
`generator.artblocks.io`) to the relevant allowlists, or (b) re-host/re-pin
all 451+ museum image assets (and any migrated videos) to IPFS/Arweave or
the existing `d3lqz0a4bldqgf.cloudfront.net` distribution before publish.
Given the volume (2,276 raw URI references, 451 distinct image assets),
option (b) is a substantial data-migration project in its own right and
should be scoped separately from the CMS package conversion.

## Protocol caps

**No hard numeric caps exist in the V1 protocol** — there is no
`MAX_PAGES`, `MAX_BLOCKS_PER_PAGE`, `MAX_BLOCKS_TOTAL`, or `MAX_ASSETS`
constant anywhere in `lib/profile-cms/protocol/v1/constants.ts`,
`schemas.ts`, or `validation.ts` (confirmed by direct inspection and cross
-checked with an independent research pass). The only limits are per-field
string/array-length constraints (page title ≤160 chars, description ≤300
chars, navigation label ≤80 chars, generic id/slug pattern ≤128 chars,
`signatures`/`storage` arrays require ≥1 entry). The 143-page, 461-block,
451-asset museum package does not exceed any of these — **no field-length
truncation was needed beyond the routine title/description truncation the
converter always applies**, and no schema-level cap is "exceeded."

That said, the schema not enforcing a cap does not mean one giant package
is the right operational shape. Practical reasons to chunk anyway:

- **Editor/tooling ergonomics.** A single 143-page, 0.54 MB JSON document
  is unwieldy to review, diff, or hand-edit; the existing builder UI
  (`lib/profile-cms/builder/package.ts`) is designed around single-page
  "homepage" or small "wallet_gallery" packages, not 143-page sites.
  `getBuilderFieldIdForIssuePath` in that file assumes a single-digit page
  index (`/payload/pages/0/...`) pattern for wiring validation issues back
  to form fields — it was not built with a 143-page package in mind.
- **Blast radius of a single publish.** Every profile CMS package is
  published as one atomic, signed, hashed unit
  (`withComputedCmsHashes`/`validateCmsPackageV1` operate over the whole
  package). Bundling all 143 pages together means any single edit
  (fixing one genesis sub-page's caption, say) requires re-signing and
  re-publishing the entire 143-page/451-asset package.
- **Natural site boundaries already exist.** The legacy site itself is
  organized into 29 independent wing "exhibits" that don't cross-reference
  each other's content (each museum wing page is self-contained; the only
  fan-out is the museum root index page linking to all wings, and
  `genesis`/`6529-fund-szn1` linking to their own sub-pages).

### Recommended chunking strategy

Chunk by wing, not by an arbitrary page-count threshold, since the wings
are already the natural content boundary and are very unevenly sized (65 +
50 pages in two wings vs. 1 page in each of the other 27):

1. **`6529museum` (hub package, ~1-3 pages):** the museum root index page
   plus a navigation/index page per wing, each linking out via `page_id`
   navigation items or `button_link` blocks to the wing's own package (CMS
   packages can link across profile namespaces via safe absolute
   `https://6529.io/...` URLs even though they can't literally route
   through the reserved `/museum` root).
2. **`6529museum-genesis` (large package, 65 pages / 142 blocks):** the
   `genesis/` wing gets its own package given its size (~45% of all
   museum pages).
3. **`6529museum-fund-szn1` (large package, 50 pages / 215 blocks):**
   likewise for `6529-fund-szn1/`, the second-largest wing and the
   heaviest by block count.
4. **`6529museum-wings-misc` (small combined package, ~27 pages):** the
   remaining single-page wings (`1-of-1-art`, `6529-fam`, `ac-museum`,
   `ack-bar`, etc.) are small enough (0-19 blocks each) to combine into one
   or two mid-sized packages grouped by theme (e.g. "1/1 art & photography"
   vs. "community/fam" vs. "metaverse/institutions") rather than one
   package per tiny wing, to avoid 27 near-empty packages.

This keeps every package well under any realistic editor/UX comfort
threshold (tens of pages, not 143), isolates the two genuinely large wings
so editing one doesn't force re-publishing the other, and mirrors the
site's existing information architecture instead of an arbitrary split.

## Unconverted patterns (warnings)

The converter emits a warning — and never silently drops the underlying
content — for three categories, all confirmed by rerunning the converter
against the manifest:

| Warning code | Count | Pages affected | Meaning |
|---|---|---|---|
| `block.unconverted_video_embed` | 161 | 34 | A `<video><source src="https://videos.files.wordpress.com/..."/></video>` self-hosted video was found. Not converted to a `video` CMS block because V1's heavy-media validation rule (`validateHeavyMediaBlock`) requires a `poster_asset_id` or `interactive_policy.fallback_asset_id`, which the raw markup doesn't reliably provide (no poster image is consistently co-located with the video markup) — building one from guesswork risked emitting an incorrect/fragile block silently. |
| `block.unconverted_iframe_embed` | 11 | 3 (all in `6529-fund-szn1`: `gazers`, `jiometory-no-compute`, `non-either`) | An `<iframe src="https://generator.artblocks.io/...">` Art Blocks live-generator embed was found. V1 has an `html_embed` block type with a `sandboxed_embed` hydration policy for exactly this case, but wiring it up correctly (sandbox permission scoping, fallback asset) needs a deliberate design decision per-embed rather than a blind regex conversion. |
| `page.empty_content` | 6 | 6 (`1-of-1-art`, `6529-fam`, `6529-general-assembly`, `generative-art`, `nft-photography`, `the-institutions`) | The page's `.post-content` container is genuinely empty in the source (`<div className="post-content"></div>`) — these are stub/placeholder pages in the legacy site itself, not a converter limitation. They convert to a CMS page with zero blocks; a follow-up would need real content authored for them, migration can't invent it. |

No page triggered the generic `page.unrecognized_structure` fallback (the
"can't find any heading/paragraph/image/button, so dump raw stripped text
into one rich_text block" case) — every non-stub museum page matches the
converter's known structural patterns closely enough to decompose properly,
which is a strong signal the recurring-TSX-pattern assumption behind this
tool holds across the wider museum tree, not just the sample pages read
during design.

## Recommendation summary

1. **Structural conversion is solid**: 100/143 pages (70%) convert
   cleanly with zero warnings using the same heading/rich_text/image/
   gallery/button_link block set validated end-to-end on the capital
   pilot; the remaining 30% are fully accounted for (video/iframe embeds
   or genuinely empty source pages), not silently lost.
2. **Asset hosting is the real blocker for a production museum
   migration**, not the CMS schema: every image (451 assets, 2,276 raw
   references) lives on `dnclu2fna0b2b.cloudfront.net`, which is outside
   the current `mediaProxy.ts` allowlist. This needs an explicit decision
   (extend the allowlist vs. re-host to IPFS/Arweave/the existing
   `d3lqz0a4bldqgf.cloudfront.net`) before museum content can safely go
   live.
3. **Video/iframe support is a scoped follow-up**, not a blocker for
   piloting: 172 embeds (161 video + 11 iframe) across 37 pages need a
   deliberate `video`/`html_embed` block-building strategy (poster asset
   sourcing, sandbox policy) that this pass intentionally left as a
   tracked warning rather than a guessed block.
4. **Chunk by wing** (`genesis`, `6529-fund-szn1`, and one or two combined
   packages for the remaining 27 small wings) rather than one 143-page
   package, for editor ergonomics and publish blast-radius reasons — not
   because any protocol cap is exceeded.
